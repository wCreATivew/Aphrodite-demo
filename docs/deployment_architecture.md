# 部署架构规划 - 本地 vs 云端

**核心原则：** 低延迟、高隐私的组件本地部署，高算力、弹性需求的组件云端部署。

## 🏠 本地部署组件（低延迟 + 隐私）

### 1. 记忆系统 ⭐

**技术栈：**
- SQLite (文本存储)
- FAISS (向量检索)
- sentence-transformers (嵌入)

**硬件需求：**
- 内存：2-4GB (FAISS 索引)
- 存储：10-50GB (记忆数据库)
- CPU: 中端即可

**延迟要求：**
- 检索：<50ms
- 写入：<20ms

**为什么本地：**
1. **隐私敏感** - 用户个人记忆
2. **高频访问** - 每次对话都要检索
3. **低延迟** - 不能接受网络延迟

---

### 2. 人格核心 ⭐

**技术栈：**
- Python 规则引擎
- 轻量级状态机

**硬件需求：**
- 内存：<500MB
- CPU: 任意

**延迟要求：**
- 决策：<10ms

**为什么本地：**
1. **实时响应** - 决定角色行为
2. **无需联网** - 纯规则计算
3. **隐私** - 角色人格设定

---

### 3. 关系网络

**技术栈：**
- SQLite (关系数据库)
- 图算法库 (networkx)

**硬件需求：**
- 内存：<1GB
- 存储：<5GB

**延迟要求：**
- 查询：<20ms
- 更新：<50ms

**为什么本地：**
1. **实时状态** - 关系随时变化
2. **频繁更新** - 每次互动都影响
3. **隐私** - 用户与角色关系

---

### 4. 情绪状态

**技术栈：**
- 内存数据库 (Redis 或 dict)
- 状态追踪算法

**硬件需求：**
- 内存：<200MB

**延迟要求：**
- 读取：<5ms
- 更新：<10ms

**为什么本地：**
1. **持续追踪** - 每秒更新
2. **极低延迟** - 影响即时反应
3. **临时状态** - 不需要持久化

---

### 5. TTS 语音合成 ⭐

**技术栈：**
- GPT-SoVITS (Tier 1 角色)
- VITS (Tier 2/3 角色)

**硬件需求：**
- GPU: RTX 3060+ (推荐)
- 显存：6-12GB
- 内存：4-8GB

**延迟要求：**
- 合成：<500ms/句
- 流式：<100ms/片段

**为什么本地：**
1. **实时输出** - 对话不能等
2. **隐私** - 对话内容不上传
3. **成本** - 云端 API 贵

**优化方案：**
- Tier 1 角色用 GPU 加速
- Tier 2/3 角色用 CPU 推理
- 缓存常用语句

---

### 6. STT 语音识别

**技术栈：**
- Whisper (本地版)
- VAD (语音活动检测)

**硬件需求：**
- GPU: 可选 (加速)
- CPU: 中端

**延迟要求：**
- 识别：<300ms

**为什么本地：**
1. **实时输入** - 说话即转文字
2. **隐私** - 语音不上传
3. **离线可用** - 无网络也能用

---

## ☁️ 云端部署组件（高算力 + 弹性）

### 1. LLM 推理

**技术栈：**
- Qwen3.5 / GPT-4
- vLLM (推理加速)

**硬件需求：**
- GPU: A100/H100
- 显存：80GB+

**延迟要求：**
- 响应：1-3s (可接受)

**为什么云端：**
1. **高算力** - 本地跑不动大模型
2. **成本效益** - 按需付费
3. **弹性扩展** - 高峰期自动扩容

**优化方案：**
- 本地缓存常用回复
- 小模型处理简单对话
- 流式输出减少等待

---

### 2. 世界状态管理

**技术栈：**
- PostgreSQL (主数据库)
- Redis (缓存)

**硬件需求：**
- 内存：8-16GB
- 存储：100GB+

**延迟要求：**
- 异步更新 (秒级)

**为什么云端：**
1. **离线计算** - 用户不在时也运行
2. **大数据** - 世界状态庞大
3. **多用户同步** - 共享世界

---

### 3. 事件系统

**技术栈：**
- Celery (任务队列)
- Redis Streams

**硬件需求：**
- 内存：2-4GB

**延迟要求：**
- 异步 (分钟级)

**为什么云端：**
1. **时间推进** - 24/7 运行
2. **后台任务** - 不紧急
3. **批量处理** - 定期触发

---

### 4. 模型训练

**技术栈：**
- PyTorch
- Bert-VITS2 训练脚本

**硬件需求：**
- GPU: A100 (多卡)
- 显存：80GB+

**延迟要求：**
- 异步 (小时级)

**为什么云端：**
1. **高算力** - 训练需要大量 GPU
2. **批量处理** - 一次性训练多个角色
3. **成本效益** - 按需租用

## 🔄 本地 - 云端同步机制

### 数据同步策略

```
┌─────────────────┐          ┌─────────────────┐
│   本地 SQLite    │  异步    │   云端 PostgreSQL │
│   (主数据库)    │ ───────→ │   (备份数据库)   │
│                 │ ←─────── │                 │
│   实时读写      │  同步    │   灾备恢复       │
└─────────────────┘          └─────────────────┘
```

**同步频率：**
- 记忆数据：每次写入后异步备份
- 关系数据：每小时同步
- 世界状态：每天同步
- 模型文件：训练完成后下载

**同步工具：**
- SQLite → PostgreSQL: ETL 脚本
- 文件同步：rclone / rsync
- 增量备份：WAL 日志

## ⚠️ 风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| **本地 GPU 故障** | TTS 不可用 | 云端 TTS 备用 |
| **云端 LLM 断连** | 对话中断 | 本地小模型 fallback |
| **数据同步失败** | 数据丢失 | WAL 日志 + 定期备份 |
| **隐私泄露** | 用户数据外泄 | 本地加密 + 云端脱敏 |
| **成本超支** | 预算不足 | 本地化更多组件 |
